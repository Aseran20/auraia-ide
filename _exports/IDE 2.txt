❯ Et pourquoi tu fais des patches et tu vas pas juste modifier le code source   
  en lui-même pour comprendre ?                                                 
● Bonne question. Les patches existent parce que le dossier vscode/ n'est pas à
  nous — c'est un clone temporaire de microsoft/vscode que le build script
  re-télécharge à chaque build. Si je modifie directement les fichiers dans
  vscode/, ils sont écrasés au prochain build.sh.

  Le flow VSCodium c'est :
  1. get_repo.sh clone un VS Code frais
  2. prepare_vscode.sh applique les patches par-dessus
  3. Gulp compile le résultat

  Les patches dans patches/user/ sont notre seul moyen de modifier le code VS
  Code de façon persistante. C'est comme des instructions "à chaque build,
  applique ces changements". Le code source original reste intact dans le repo
  Microsoft, et nos modifications sont isolées dans des fichiers .patch qu'on
  contrôle.

  C'est le même modèle que Cursor, Windsurf, et VSCodium eux-mêmes — ils ne
  forkent pas directement le code VS Code, ils maintiennent des patches
  appliqués au build.
